1. System Overview (Tổng quan hệ thống)

Mô tả hệ thống

Hệ thống cung cấp trải nghiệm thuyết minh tự động (Auto-narration) qua thiết bị di động cho du khách khi tham quan phố ẩm thực Vĩnh Khánh. Dựa trên vị trí GPS (Geofencing), ứng dụng sẽ tự động phát các đoạn audio giới thiệu về lịch sử, văn hóa, và các món ăn đặc trưng của từng quán/khu vực khi du khách đi ngang qua.

Problem (Vấn đề)

  • Phố Vĩnh Khánh ồn ào, đông đúc, du khách nước ngoài khó tiếp cận thông tin văn hóa ẩm thực.
  • Thiếu hướng dẫn viên tại chỗ hoặc rào cản ngôn ngữ với chủ quán.
  • Trải nghiệm xem bản đồ truyền thống khiến du khách mất tập trung vào cảnh quan thực tế.

Goal (Mục tiêu)

  • Tạo ra một "Virtual Tour Guide" rảnh tay (Hands-free experience).
  • Cung cấp nội dung đa ngôn ngữ (Anh, Việt, Hàn, Nhật, Trung).
  • Hoạt động ổn định ngay cả khi kết nối mạng kém (Offline-first approach).

3. System Flow (Luồng hệ thống / Logic cốt lõi)

Luồng xử lý từ lúc khởi động app đến khi phát audio tự động. Yêu cầu khắt khe về việc xử lý trùng lặp và ưu tiên phát.

Bước 1: App Initialization & Data Sync

App khởi động. Gọi API load toàn bộ metadata POI và Geofence data từ Server. Lưu trữ cục bộ vào SQLite. User chọn "Download Tour Audio" để tải file .mp3 về local storage.

Bước 2: Background GPS Tracking

Hệ thống kích hoạt Service chạy ngầm. Thu thập tọa độ GPS mỗi 3-5 giây. Sử dụng thuật toán Haversine hoặc Native Geofencing API để tính khoảng cách đến các điểm POI.

Bước 3: Geofence Trigger Event

Tọa độ GPS rơi vào bán kính Geofence (VD: 15m xung quanh quán Ốc Oanh). Event OnGeofenceEnter được kích hoạt.

// Pseudocode Trigger Logic
IF (Distance(User, POI) <= POI.Radius) THEN
    TriggerEvent(POI_ID)
END IF
Bước 4: Validation & Playback Logic (CRITICAL)

Khi event được trigger, hệ thống phải kiểm tra điều kiện khắt khe trước khi phát để tránh spam audio:

FUNCTION HandleGeofenceEnter(POI_ID)
    // 1. Kiểm tra lịch sử phát
    IF (Session.PlayedPOIs.Contains(POI_ID) AND TimeSinceLastPlay < 24_HOURS) THEN
        RETURN // Bỏ qua, tránh phát lại liên tục khi user đi vòng quanh
    END IF

    // 2. Kiểm tra trạng thái Audio Engine
    IF (AudioPlayer.IsPlaying) THEN
        CurrentPOI = AudioPlayer.GetCurrentPOI()
        
        // 3. Xử lý Priority (Ưu tiên)
        IF (POI_ID.Priority > CurrentPOI.Priority) THEN
            AudioPlayer.Stop() // Ngắt bài cũ (ví dụ: Thông báo khẩn cấp/điểm chính yếu)
            AudioQueue.Clear()
        ELSE
            AudioQueue.Enqueue(POI_ID) // Cho vào hàng đợi
            RETURN
        END IF
    END IF

    // 4. Lấy dữ liệu và Phát
    AudioFile = SQLite.GetAudio(POI_ID, CurrentLanguage)
    AudioPlayer.Play(AudioFile)
    
    // 5. Ghi nhận lịch sử
    Session.PlayedPOIs.Add(POI_ID)
    LogAnalyticsAsync(POI_ID, EventType="AutoPlay_Started")
END FUNCTION

4. Architecture (Kiến trúc hệ thống)

Hệ thống được thiết kế theo kiến trúc Client-Server với nguyên lý Offline-First tại phía Client.

Công nghệ sử dụng

  • Mobile App: .NET MAUI (C#). Chạy đa nền tảng iOS/Android từ một codebase. Tích hợp native GPS dễ dàng.
  • Backend API: ASP.NET Core 8 Web API. Xử lý logic nghiệp vụ, xác thực, trả về JSON.
  • Database (Server): SQL Server. Lưu trữ toàn bộ dữ liệu hệ thống, phân quyền, Analytics.
  • Local Storage (Client): SQLite-net. Lưu POI metadata và trạng thái app offline.
  • File Storage: Azure Blob Storage hoặc AWS S3 (Lưu audio files).

Tại sao cần Offline-First?

Khu vực phố Vĩnh Khánh rất đông người vào buổi tối, mạng 4G/5G thường xuyên bị nghẽn. Du khách quốc tế có thể không có SIM 4G data lớn. Việc tải trước Audio và lưu tọa độ Geofence xuống SQLite giúp app hoạt động trơn tru, không độ trễ, và không phụ thuộc kết nối mạng lúc đang trải nghiệm.

Khi nào dùng Server vs Local?

  • Local (SQLite): Trigger logic, đọc nội dung POI, phát audio.
  • Server (API): Sync dữ liệu ban đầu, gửi Analytics log (âm thầm khi có mạng), tải bản cập nhật nội dung.

Thành phần & Luồng dữ liệu: Sơ đồ mô tả logic tuần tự khi có event GPS. Đi qua các block quyết định (If/Else) về trạng thái phát, trạng thái hệ thống audio hiện tại, và ưu tiên (Priority). Cuối cùng kết thúc bằng việc Play và Log.

Tại sao thiết kế vậy: Logic Check Priority rất quan trọng ở phố ẩm thực vì các POI nằm sát nhau. Nếu không có hàng đợi (Queue) hoặc ngắt (Stop), nhiều audio sẽ đè lên nhau gây nhiễu. "Already Played" giúp tránh spam khi user đứng yên hoặc đi lặp lại một đoạn đường.

6. Use Cases (Ca sử dụng)

👨‍🦯 Tourist (Du khách)

Mô tả: Khám phá phố Vĩnh Khánh, nghe thuyết minh.

Trigger: Mở app, tải tour, cất điện thoại vào túi.

Flow: App theo dõi ngầm -> Tới khu ốc -> Tự động phát audio "Lịch sử ngõ ốc Vĩnh Khánh" tiếng Anh -> Tới quán Ốc Oanh -> Phát "Món ốc hương xào bơ cay đặc sản" -> User nghe xong, app tự gửi log thời gian nghe.

👨‍🍳 Shop Owner (Chủ quán)

Mô tả: Đóng góp nội dung cho quán của mình.

Trigger: Đăng nhập CMS (Phân quyền Owner).

Flow: Chọn POI quán mình -> Cập nhật Text giới thiệu món mới -> Yêu cầu Admin record Audio hoặc tự upload file mp3 -> Xem thống kê: Bao nhiêu người đã đi qua và nghe audio quán mình tháng này.

👨‍💻 Admin (Hệ thống)

Mô tả: Quản trị toàn bộ nền tảng.

Trigger: Đăng nhập Dashboard.

Flow: Vẽ Geofence (Polygon/Circle) trên bản đồ cho POI mới -> Gắn file Audio đa ngôn ngữ -> Xem Heatmap để biết luồng đi của khách du lịch tập trung ở đầu phố hay cuối phố.

Thiết kế tập trung vào tính toàn vẹn dữ liệu và khả năng truy xuất đa ngôn ngữ nhanh chóng.


[Users]          [POIs]                 [Tours]

- Role           - Lat, Lng             - Name

                 - Priority

- Name                   - FileSize
- ContentHtml            - DurationSec
- Unique(POIId, LangCode)

[UserSessions]            [ListeningHistory]
- Id (PK)                 - Id (PK)
- DeviceId                - SessionId (FK)
- StartTime               - POIId (FK)
- EndTime                 - LangCode
                          - ListenedSec
                          - TriggerType ('GPS' or 'QR')
                          - CreatedAt
Quan hệ bảng & Tại sao thiết kế vậy:
  • Translations nối với POIs (1-N). Ràng buộc Unique Key trên (POIId, LangCode) đảm bảo mỗi POI chỉ có duy nhất 1 bản dịch tiếng Anh, 1 tiếng Việt...
  • AudioFiles tách riêng khỏi Translation giúp tái sử dụng text nhưng có thể update file audio chuẩn chất lượng cao sau.
  • ListeningHistory thu thập dữ liệu thô (Raw Event Data) lưu trữ từng sự kiện nghe của user. Định danh qua SessionId (tạo random mỗi lần mở app) thay vì UserId để bảo vệ quyền riêng tư của du khách (Không yêu cầu đăng nhập).

Các API giao tiếp giữa Mobile App và .NET Core Backend.

{
  "tourId": "uuid",

  "pois": [
    {

      "lat": 10.7612, "lng": 106.7021,
      "radius": 20, "priority": 1,

         "vi": { "title": "...", "audioUrl": "https://..." },

Sử dụng: Gọi 1 lần duy nhất khi app khởi động hoặc khi ấn "Update Content". Client tự so sánh version để quyết định tải hay không.

POST /api/v1/analytics/bulk-log 
// Request JSON (App gửi ngầm định kỳ)
{
  "sessionId": "ses_987",
  "events": [
    { "poiId": "poi_123", "action": "AUTO_PLAY_FINISH", "timestamp": "2023-10-01T19:00Z" },
    { "poiId": "poi_456", "action": "MANUAL_QR_PLAY", "timestamp": "2023-10-01T19:15Z" }
  ]
}

Sử dụng: App MAUI sẽ lưu cache các event này vào SQLite. Khi có mạng WiFi hoặc 4G ổn định, Background Worker sẽ push cục (bulk) lên server để tiết kiệm pin và băng thông. API này nạp dữ liệu cho Heatmap và Reports.

9. Core Components (Thành phần cốt lõi)

📍 Geofence Engine (Native / Haversine)

Thành phần chịu trách nhiệm tính toán khoảng cách. Do phố Vĩnh Khánh khá hẹp, độ chính xác GPS là yếu tố sống còn.

  • Cơ chế: Không dùng Google Maps API liên tục (tốn phí & pin). Sử dụng Geolocator native của iOS/Android qua MAUI Essentials.
  • Công thức: Tính khoảng cách giữa User (Lat1, Lng1) và POI (Lat2, Lng2) bằng công thức Haversine cục bộ.
  • Tối ưu pin: Dynamic interval. Nếu user ở xa phố Vĩnh Khánh > 5km, ping 10 phút/lần. Nếu cách < 1km, ping 1 phút/lần. Đi vào trong phố: ping 5 giây/lần.

🎧 Narration Engine (Queue Manager)

Bộ não điều phối âm thanh.

  • State Machine: Có các trạng thái Idle, Playing, Paused (khi có cuộc gọi đến).
  • Deduplication Logic: Duy trì một HashSet trong memory chứa các POI đã phát trong session hiện tại. Nếu ID đã tồn tại -> Reject play.
  • Queueing: Dùng ConcurrentQueue. Nếu user chạy xe máy nhanh qua 3 quán, thay vì phát 3 bài cùng lúc, nó nhét vào hàng đợi và phát bài của quán ưu tiên cao nhất, bỏ qua các quán nhỏ.

10. CMS & Analytics (Quản trị & Phân tích)

Heatmap & Tracking

Dữ liệu từ API /bulk-log được SQL Server xử lý và trực quan hóa lên màn hình Dashboard bằng thư viện bản đồ (vd: Mapbox/Leaflet).

Logic: Mỗi khi event GPS được ping (kể cả không phát audio), hệ thống lưu tọa độ. CMS gom nhóm (Clustering) các điểm này. Chỗ nào mật độ cao -> hiển thị màu đỏ (Hotspot). Giúp Ban quản lý phố biết khu vực nào hay bị tắc nghẽn hoặc thu hút khách nhất.

11. UI/UX (Giao diện & Trải nghiệm)

Màn hình chính: Map-Centric Design

Khác với app tin tức, app audio guide mở ra là hiển thị bản đồ toàn màn hình. Các POI hiển thị dưới dạng Icon trực quan (Biểu tượng dĩa ốc, ly bia...).

BottomSheet Component

Khi một Audio đang phát, một "BottomSheet" (bảng nhỏ) tự động trượt lên từ cạnh dưới màn hình hiển thị: Tên quán, Ảnh minh họa, Thanh tiến trình audio (Progress bar), Nút Pause/Play, Nút Read Text (cho người khiếm thính).

Settings (Rất quan trọng)

  • GPS Sensitivity: Cho phép user chỉnh bán kính trigger (Gần/Vừa/Xa). Mặc định là 15m.
  • Auto-play Toggle: Cho phép tắt phát tự động, chuyển sang chế độ "Tap to play" (Chạm vào icon trên map để nghe).
  • Offline Mode: Nút "Download All Assets (50MB)" hiển thị rõ ràng trước khi bắt đầu.

14. Non-Functional (Yêu cầu phi chức năng)

⚡ Performance

- API Sync Response Time < 200ms.

- Geofence Trigger Delay < 1.5 giây (Từ lúc bước vào vùng đến lúc nghe tiếng).

- Kích thước App tải về < 30MB (Không tính content tải thêm).

🔒 Security & Scalability

- Security: App giao tiếp API qua HTTPS. JWT Auth cho Admin CMS. Dữ liệu tracking khách du lịch ẩn danh hoàn toàn (Không lưu sdt, email).

- Scalability: Backend thiết kế Stateless, có thể scale ngang (Horizontal scaling) qua Kubernetes/Docker khi có lễ hội ẩm thực lượng khách tăng đột biến.

15. Conclusion (Tổng kết & Hướng phát triển)

Hệ thống Thuyết minh tự động cho Phố ẩm thực Vĩnh Khánh (MVP) tập trung giải quyết bài toán cốt lõi: Trải nghiệm rảnh tay, không giới hạn ngôn ngữ, và hoạt động tốt ở môi trường mạng kém. Kiến trúc .NET MAUI và SQLite là sự lựa chọn tối ưu cho yêu cầu Offline-first này.
